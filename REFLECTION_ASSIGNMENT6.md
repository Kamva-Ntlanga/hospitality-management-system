# Assignment 6 Reflection: Challenges in Agile Planning

## Introduction

This assignment required translating system requirements and use cases from Assignments 4 and 5 into user stories, creating a prioritized backlog, and planning an initial sprint. As the sole person playing multiple Scrum roles, I faced several internal challenges in making objective decisions.

## Challenge 1: Prioritizing Without Real Stakeholders

In a real Agile team, the Product Owner represents stakeholders. Here, I had to imagine stakeholder perspectives and make trade-off decisions alone. I found myself arguing with my own assumptions about what is "must-have" versus "should-have."

**How I addressed it:** I created a framework based on stakeholder success metrics from Assignment 4. Features enabling core stakeholder jobs became Must-have, efficiency improvements became Should-have, and convenience features became Could-have.

## Challenge 2: Estimating Story Points Alone

Agile teams use planning poker for collaborative estimation. As one person, I had no one to challenge my estimates or point out blind spots.

**How I addressed it:** I used reference-based estimation, comparing each story to known baselines. A simple form field is 2 points; a full booking flow with validation and email is 5 points. I also added a 20% buffer for unknown complexity.

## Challenge 3: Balancing Technical and User Stories

I had to decide whether to include US-016 (encryption) in the first sprint. Users don't see encryption, but it is critical for security compliance from Assignment 4.

**How I addressed it:** I included it as Must-have because security is non-negotiable, stakeholders require zero security breaches, and retrofitting security after development is more expensive.

## Challenge 4: Defining the Sprint Goal

The sprint goal needed to be specific enough to guide the team but flexible enough to allow negotiation. My first few versions were either too vague or too detailed.

**How I addressed it:** I settled on a goal focused on core value: allowing guests to complete a booking from search to payment, and enabling staff to serve walk-in guests. This excludes Should-have features to keep the sprint achievable.

## Challenge 5: Saying No to Features

US-011 (OTA integration with Booking.com) is valuable for the Marketing Team but technically complex.

**How I addressed it:** I deferred it to Won't-have for the initial release. The justification: OTA integration requires external API agreements, testing with real OTAs, and ongoing maintenance. It can be added in a future sprint after core booking is stable.

## Conclusion

This assignment showed that Agile planning requires objective prioritization, realistic estimation, and clear goal definition. The frameworks of MoSCoW prioritization, story points, and sprint goals provide structure, but discipline and objectivity are what make them work. These skills will translate directly to real-world Agile teams.
